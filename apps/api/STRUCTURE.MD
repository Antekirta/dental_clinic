# API Structure

Этот файл фиксирует принципы организации `apps/api`, чтобы дальше развивать API последовательно, а не собирать его как набор случайных роутов и сервисов.

## Цель структуры

Структура строится:

- по доменам, а не по абстрактным слоям;
- с тонкими контроллерами;
- с явным service-layer для use case;
- с возможностью вызывать API из `n8n`, админки и внутренних сценариев одинаково.

Главная идея: orchestration и бизнес-правила должны жить в Python-коде FastAPI, а не в SQL-нодах `n8n`.

## Текущая структура

```text
app/
  api/
    router.py
    routes/
  db/
    base.py
    session.py
    models/
  modules/
    contacts/
      api.py
      schemas.py
      service.py
    inbound_messages/
    conversations/
    appointment_requests/
    appointments/
    handoff_tasks/
    reference_data/
```

## Правила по папкам

### `app/api`

Здесь лежит только верхнеуровневая сборка API:

- общий `api_router`;
- legacy/simple routes вроде `health`;
- подключение доменных роутеров.

`app/api` не должен содержать основную бизнес-логику.

### `app/modules/<domain>`

Каждый домен оформляется отдельным модулем.

Минимальный состав модуля:

- `api.py` — HTTP controller / router;
- `schemas.py` — request/response модели;
- `service.py` — use case и бизнес-логика.

Если модуль разрастётся, в него можно добавлять:

- `selectors.py` — сложные read/query-функции;
- `constants.py` — константы домена;
- `errors.py` — доменные исключения;
- `validators.py` — доменные проверки;
- `orchestration.py` — если внутри домена появятся сложные сценарии.

### `app/db/models`

SQLAlchemy-модели остаются общей картой БД и не дробятся по доменам до тех пор, пока это реально не начнет мешать.

## Правила по слоям

### Контроллеры (`api.py`)

Контроллер должен быть тонким.

Он:

- принимает HTTP-запрос;
- валидирует payload через Pydantic;
- вызывает service-функцию;
- преобразует доменные ошибки в HTTP-ответы.

Контроллер не должен:

- содержать SQL;
- принимать бизнес-решения;
- дублировать логику из сервиса;
- координировать несколько доменов вручную.

### Схемы (`schemas.py`)

Схемы описывают контракт API:

- что приходит;
- что уходит;
- какие поля обязательны;
- какие поля можно игнорировать.

Для интеграционных endpoint’ов допустимо использовать `extra="ignore"`, если endpoint должен принимать payload, похожий на нормализованный объект из `n8n`.

### Сервисы (`service.py`)

Сервис — это главное место для бизнес-логики.

Сервис:

- работает с `Session`;
- ищет сущности;
- применяет правила матчинга;
- создает и изменяет данные;
- возвращает результат use case.

Сервис не должен знать про HTTP-слой.

## Принцип: use case важнее CRUD

Мы не строим API как набор абстрактных CRUD-сервисов вида `ContactService` на все случаи жизни.

Мы строим сервисы вокруг конкретных сценариев:

- `resolve_contact`
- `resolve_or_create_contact`
- `get_or_create_conversation`
- `process_incoming_message`
- `create_handoff_task`

То есть точка входа в сервис должна отражать действие, а не просто сущность.

## Принцип: домены могут общаться, но не хаотично

Домены не обязаны быть полностью изолированными. Общение допустимо, но должно быть направленным и понятным.

Правила:

- доменные сервисы могут вызывать сервисы других доменов;
- нельзя допускать циклические зависимости;
- сложные междоменные сценарии лучше собирать в orchestration-модуле.

Хороший пример:

- `inbound_messages.service` вызывает:
  - `contacts.service`
  - `conversations.service`
  - `appointment_requests.service`
  - `handoff_tasks.service`

Плохой пример:

- `contacts.service` вызывает `conversations.service`;
- `conversations.service` вызывает `contacts.service`;
- оба начинают зависеть друг от друга по кругу.

## Роль orchestration-доменов

Некоторые домены по природе являются координаторами сценария, а не “владельцами одной таблицы”.

Для этого проекта таким доменом должен стать:

- `inbound_messages`

Именно он должен в перспективе:

1. принять нормализованный входящий payload;
2. найти или создать контакт;
3. найти или создать conversation;
4. создать message;
5. запустить дальнейшую логику по intent / booking / handoff.

## Текущий эталонный пример

Первый реальный доменный модуль — `contacts`.

Что в нем уже реализовано:

- `POST /contacts/resolve`
- request/response схемы в `contacts/schemas.py`
- сервис `resolve_contact(...)` в `contacts/service.py`

Этот endpoint решает задачу, которую раньше пытались собрать в `n8n` SQL-нодами:

- поиск по `channel + external_id`;
- fallback по `phone`;
- fallback по `email`.

Это считается правильным направлением развития: бизнес-логику переносим из workflow-конструктора в код API.

## Когда добавлять новый домен

Новый домен добавляется, если:

- у сущности появляется собственная бизнес-логика;
- нужен отдельный HTTP-контракт;
- сценарий начинает разрастаться и перестает помещаться в существующий модуль.

Не нужно создавать домен заранее “про запас”, если там пока нет реального use case.

## Как расширять структуру дальше

Рекомендуемый порядок:

1. сначала `contacts`;
2. затем `inbound_messages`;
3. затем `conversations`;
4. затем `appointment_requests`;
5. затем `handoff_tasks`;
6. затем остальные домены по мере необходимости.

## Практические правила на будущее

- Новые endpoint’ы добавлять в доменные модули, а не в `app/api/routes`.
- Новую бизнес-логику сначала пытаться выразить через сервис, а не через `n8n` SQL.
- Если логика поиска или создания нужна в нескольких местах, выносить ее в service-layer.
- Если в сервисе становится слишком много SQL-чтения, выносить query-функции в `selectors.py`.
- Если сценарий координирует несколько доменов, делать orchestration-service, а не разносить куски логики по контроллерам.

## Важная оговорка

Этот файл — не догма. Если структура начнет мешать, ее можно менять.

Но по умолчанию все новые изменения в `apps/api` должны соответствовать этим принципам, пока не появится явная причина их пересмотреть.
