🦷 BrightSmile Dental Clinic

A platform for automated processing of inbound patient messages via Telegram, with intent classification powered by Google Gemini AI and smart routing to human operators.

Python 3.11 FastAPI + SQLAlchemy 2.0 Gemini 2.0 Flash PostgreSQL Telegram Bot API

Project Description

BrightSmile is a backend platform for a dental clinic that automatically receives patient messages via Telegram, classifies their intent using Gemini AI, auto-replies to common questions, and escalates complex requests to human administrators.

What the System Does

1 Patient sends a message to the clinic's Telegram bot
2 Webhook delivers the update to the FastAPI server
3 Contact is automatically resolved or created in the database
4 Gemini AI classifies the intent and extracts entities (name, date, service…)
5 The system routes the message: auto-reply, data collection, or escalation
6 If auto-reply is needed — Gemini generates text, the bot sends it to the patient
7 Everything is saved to the DB: message, intent, conversation status, admin task

Two-Call Gemini Design

Every inbound message is processed by two sequential Gemini API calls:

CallFunctionInputOutput
Call 1 classify_message() text + history + contact context intent_code, route_type, confidence, entities
Call 2 generate_reply() intent + entities + reference data reply text for the patient

Both calls use structured JSON output (response_mime_type: application/json) — no free-form text parsing required.

Telegram Bot

Inbound Flow

Telegram delivers updates via webhook to the POST /webhooks/telegram endpoint. The server always returns 200 OK to prevent Telegram from retrying delivery.

The adapter normalize_telegram_update() converts a raw Telegram Update into a unified UnifiedIncomingMessage object:

Telegram message typemessage_type in systemProcessing
texttextFull processing + Gemini
photo / document with captionimageFull processing (text from caption)
sticker, voice, videounsupportedLogged, Gemini not called
channel_post, callback_query—TelegramAdapterError, silently skipped

Outbound Flow

The telegram_client.py module sends replies via the standard Telegram Bot API without any external dependencies (uses urllib).

POST https://api.telegram.org/bot{TOKEN}/sendMessage
{"chat_id": "123456789", "text": "Hello! How can I help you?"}

On failure, a TelegramClientError is raised and caught in service.py — outbound messages are not stored in the DB if delivery failed.

Registering the Webhook

# Set the webhook (run once)
curl -X POST "https://api.telegram.org/bot{TOKEN}/setWebhook" \
  -d "url=https://api.dental-clinic.kiremma.dev/webhooks/telegram"

# Verify status
curl "https://api.telegram.org/bot{TOKEN}/getWebhookInfo"

Gemini AI

Client Initialisation

The client is created lazily on first use (deferred initialisation), allowing tests to mock it before any module import:

from google import genai
from google.genai import types

_client: genai.Client | None = None

def _get_client() -> genai.Client:
    global _client
    if _client is None:
        _client = genai.Client(api_key=settings.gemini_api_key)
    return _client

Classify Call — Intent Classification

The system prompt describes all 28 allowed intent_code values and 8 extractable entities. The response is forced to JSON via response_mime_type:

response = _get_client().models.generate_content(
    model=settings.gemini_model,
    contents=CLASSIFICATION_SYSTEM_PROMPT + "\n\n" + user_prompt,
    config=types.GenerateContentConfig(
        response_mime_type="application/json",
        response_schema={
            "type": "object",
            "properties": {
                "intent_code":        {"type": "string"},
                "confidence":         {"type": "number"},
                "extracted_entities": {"type": "object", ...}
            },
            "required": ["intent_code", "confidence"]
        },
        temperature=0.1,  # low temperature for consistency
    )
)

Extractable Entities

name phone email service date_preference time_preference doctor_preference promotion_name

Reply Call — Response Generation

Called only for auto_reply and auto_reply_and_collect routes. System prompt rules: 1–3 sentences, one clear next step, same language as the patient.

response = _get_client().models.generate_content(
    model=settings.gemini_model,
    contents=REPLY_SYSTEM_PROMPT + "\n\n" + user_prompt,
    config=types.GenerateContentConfig(
        temperature=0.4,
        max_output_tokens=500,
    )
)

The user_prompt for reply generation includes:

  • The classified intent
  • Extracted entities
  • Missing fields for booking (if data collection is ongoing)
  • Clinic reference data (hours, address, or prices — depending on intent)
  • The last 10 messages of conversation history
⚠️ API Quota: The system uses gemini-2.0-flash. The free tier may be exhausted under active use. For production, enabling billing in Google AI Studio is recommended.

Routing Logic

After classification, Python code (not Gemini) makes the routing decision based on INTENT_ROUTE_MAP:

RouteActionConversation StatusHandoffTask
auto_reply Gemini generates a reply and sends it to the patient classified None
auto_reply_and_collect Bot collects booking data, asks for the next missing field waiting_for_patient_data → booking_request_created None
handoff_admin HandoffTask created, bot sends acknowledgement to patient waiting_for_admin Yes
handoff_urgent Emergency escalation, bot notifies patient urgent_handoff Yes (type: urgent_case)

Task Priorities

IntentPriority
emergencyurgent
complaint_or_negative_feedbackhigh
post_visit_followuphigh
All other handoff intentsnormal

Booking Flow

For intents in the BOOKING_INTENTS group, the system saves accumulated data to AppointmentRequest and computes which fields are still missing:

Required fields for a booking: namephoneservicedate_preference

The bot asks for one missing field at a time until all required data is collected. The conversation then transitions to booking_request_created and an administrator confirms the slot manually.
