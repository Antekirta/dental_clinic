🦷 BrightSmile Dental Clinic

Платформа автоматической обработки входящих сообщений пациентов через Telegram с классификацией интентов на базе Google Gemini AI и умной маршрутизацией на администраторов.

Python 3.11 FastAPI + SQLAlchemy 2.0 Gemini 2.0 Flash PostgreSQL Telegram Bot API

Описание проекта

BrightSmile — это backend-платформа для стоматологической клиники, которая автоматически принимает сообщения пациентов через Telegram, классифицирует их намерение с помощью Gemini AI, автоматически отвечает на типовые вопросы и эскалирует сложные запросы администраторам.

Типа маршрутизации

Что делает система

1 Пациент пишет сообщение в Telegram-бота клиники
2 Webhook доставляет обновление на сервер FastAPI
3 Контакт автоматически разрешается или создаётся в базе данных
4 Gemini AI классифицирует интент и извлекает сущности (имя, дата, услуга…)
5 Система маршрутизирует сообщение: автоответ, сбор данных или эскалация
6 Если нужен автоответ — Gemini генерирует текст, бот отправляет его пациенту
7 Всё сохраняется в БД: сообщение, интент, статус разговора, задача для администратора

Двухвызовой дизайн Gemini

Каждое входящее сообщение обрабатывается двумя последовательными вызовами Gemini API:

ВызовФункцияВходные данныеРезультат
Call 1 classify_message() текст + история + контекст контакта intent_code, route_type, confidence, entities
Call 2 generate_reply() интент + сущности + справочные данные текст ответа пациенту

Оба вызова используют структурированный JSON вывод (response_mime_type: application/json) — никакого парсинга свободного текста.

Telegram Bot

Входящий поток (Inbound)

Telegram доставляет обновления по webhook на эндпоинт POST /webhooks/telegram. Сервер всегда возвращает 200 OK, чтобы Telegram не повторял доставку.

Адаптер normalize_telegram_update() конвертирует raw Telegram Update в унифицированный объект UnifiedIncomingMessage:

Тип сообщения Telegrammessage_type в системеОбработка
texttextПолная обработка + Gemini
photo / document с подписьюimageПолная обработка (текст из подписи)
sticker, voice, videounsupportedЛогируется, Gemini не вызывается
channel_post, callback_query—TelegramAdapterError, тихий пропуск

Исходящий поток (Outbound)

Модуль telegram_client.py отправляет ответы через стандартный Telegram Bot API без внешних зависимостей (используется urllib).

POST https://api.telegram.org/bot{TOKEN}/sendMessage
{"chat_id": "123456789", "text": "Здравствуйте! Чем могу помочь?"}

При ошибке выбрасывается TelegramClientError, которая перехватывается в service.py — исходящее сообщение не сохраняется в БД, если не было доставлено.

Регистрация Webhook

# Установить webhook (выполнить один раз)
curl -X POST "https://api.telegram.org/bot{TOKEN}/setWebhook" \
  -d "url=https://api.dental-clinic.kiremma.dev/webhooks/telegram"

# Проверить статус
curl "https://api.telegram.org/bot{TOKEN}/getWebhookInfo"

Gemini AI

Инициализация клиента

Клиент создаётся лениво при первом обращении (deferred initialization), что позволяет мокировать его в тестах до импорта модуля:

from google import genai
from google.genai import types

_client: genai.Client | None = None

def _get_client() -> genai.Client:
    global _client
    if _client is None:
        _client = genai.Client(api_key=settings.gemini_api_key)
    return _client

Classify Call — Классификация интента

Системный промпт описывает 28 допустимых значений intent_code и 8 извлекаемых сущностей. Ответ принудительно в JSON через response_mime_type:

response = _get_client().models.generate_content(
    model=settings.gemini_model,
    contents=CLASSIFICATION_SYSTEM_PROMPT + "\n\n" + user_prompt,
    config=types.GenerateContentConfig(
        response_mime_type="application/json",
        response_schema={
            "type": "object",
            "properties": {
                "intent_code":        {"type": "string"},
                "confidence":         {"type": "number"},
                "extracted_entities": {"type": "object", ...}
            },
            "required": ["intent_code", "confidence"]
        },
        temperature=0.1,  # низкая температура для консистентности
    )
)

Извлекаемые сущности

name phone email service date_preference time_preference doctor_preference promotion_name

Reply Call — Генерация ответа

Вызывается только для маршрутов auto_reply и auto_reply_and_collect. Правила в системном промпте: ответ 1–3 предложения, один чёткий следующий шаг, тот же язык, что и пациент.

response = _get_client().models.generate_content(
    model=settings.gemini_model,
    contents=REPLY_SYSTEM_PROMPT + "\n\n" + user_prompt,
    config=types.GenerateContentConfig(
        temperature=0.4,
        max_output_tokens=500,
    )
)

В user_prompt для генерации ответа передаётся:

  • Классифицированный интент
  • Извлечённые сущности
  • Недостающие поля для записи (если идёт сбор данных)
  • Справочные данные клиники (часы, адрес или цены — в зависимости от интента)
  • История последних 10 сообщений в диалоге
⚠️ Квота API: Система использует gemini-2.0-flash. Бесплатный уровень может быть исчерпан при активном использовании. Для продакшена рекомендуется подключить оплату в Google AI Studio.

Логика маршрутизации

После классификации Python-код (не Gemini) принимает решение о маршруте на основе INTENT_ROUTE_MAP:

МаршрутДействиеСтатус разговораHandoffTask
auto_reply Gemini генерирует ответ и отправляет пациенту classified Нет
auto_reply_and_collect Бот собирает данные для записи, спрашивает недостающее поле waiting_for_patient_data → booking_request_created Нет
handoff_admin Создаётся HandoffTask, бот отправляет подтверждение принятия waiting_for_admin Да
handoff_urgent Экстренная эскалация, бот уведомляет пациента urgent_handoff Да (тип: urgent_case)

Booking Flow — Сценарий записи

Для интентов из группы BOOKING_INTENTS система сохраняет накопленные данные в AppointmentRequest и вычисляет недостающие поля:

Обязательные поля для записи: namephoneservicedate_preference

Бот последовательно запрашивает по одному полю за раз, пока все данные не будут собраны. Затем статус переходит в booking_request_created и администратор подтверждает запись вручную.
